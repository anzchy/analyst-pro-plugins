## Problem Statement

AnalystPro 内置了 14 个投研工作流 skills（deal-analysis、tech-dd、industry-research 等），目前只能在 AnalystPro Electron 应用内的 SkillPalette 调用。希望把其中 deal/dd/research 三类核心工作流抽出来，封装成 Claude Code Plugin，让任何 Claude Code 终端都能 `/plugin install` 后直接调用，命令带前缀（`/analyst-deal:xxx` / `/analyst-dd:xxx` / `/analyst-research:xxx`），方便和用户已有的 40+ 其他 skill 区分。

## What Makes This Worth Doing

- **复用 prompt 资产**：AnalystPro 的 SKILL.md + agent prompt 已经迭代了多个 phase，是高密度的领域知识。把它们 plugin 化等于把成熟工作流以最低摩擦的形式开放给任何 Claude Code 用户（包括投研团队同事、非工程背景的合伙人）。
- **跨终端一致性**：用户在 Mac 终端、Linux 服务器、CI 环境都能装上同一套命令，行为一致。
- **GitHub-native 分发**：marketplace 就是一个 GitHub repo，更新即推送，安装即 `gh` 拉取。
- **解耦 AnalystPro UI 依赖**：plugin 化迫使我们把"工作流逻辑"与"AnalystPro UI/Workspace"分离，副产物是 AnalystPro 自身也更模块化。

## Constraints

1. **三个独立 plugin = 三个独立 prefix**。Claude Code plugin 系统的 prefix 来自 `plugin.json` 的 `name` 字段，自动加在所有 commands 前。不能在一个 plugin 内手动声明三个 prefix。
2. **AnalystPro 私有依赖必须切断**：当前 SKILL.md 引用 `agent: <name>`（Secretary→subagent 调度）、`workspace/state/...`、`inbox/...`、`ap-*` 扩展字段、Playwright MCP 浏览器工具、Codex MCP polish 工具。这些在 Claude Code 里要么不存在、要么需要用户自行配置。
3. **`interview-notes-enricher`**\*\* 当前硬编码"矽睿"项目\*\*：必须先通用化（参数化 memo 路径 + transcript glob）才能进 plugin。
4. **不引入新的 AnalystPro 运行时依赖**：plugin 是给"裸 Claude Code"用的，不能假设用户跑了 AnalystPro。
5. **Markdown-only 工作流**：plugin 里所有产物都是 Markdown 文件，写到用户的当前工作目录或 plugin 提示让用户指定的目录。

## Premises

1. 用户的真实意图是"开放工作流给更广的 Claude Code 用户"，而不是"在 AnalystPro 内换一种调用方式"。所以 plugin **不依赖** AnalystPro 安装。
2. 工作流的核心价值在 prompt（怎么搜、搜什么、按什么顺序）+ knowledge files（行业地图、DD checklist、IC memo 模板）。**这两类资产必须同时进 plugin**，否则只搬 prompt 用户就缺了模板。
3. Subagent 不必单独定义。每个命令文件 `commands/<name>.md` 内联完整工作流描述，让 Claude（运行 plugin 的那个 model）直接执行。Plugin 不需要 `agents/` 子目录。**这是和 AnalystPro 最大的架构差异**——AnalystPro 是 Secretary 调度 subagent，plugin 里就是单 Claude 直接干。
4. Playwright MCP 和 Codex MCP 通过 plugin 自带的 `.mcp.json` 声明，用户安装 plugin 后由 Claude Code 自动加载。但仍需用户本机有 `npx`（Playwright）或 `codex` CLI 已登录。
5. Workspace 路径默认用 `./workspace/`（相对当前工作目录）。每个命令第一次运行时检测目录是否存在，若不存在则 AskUserQuestion 让用户选：在 cwd 下创建 / 指定其他路径 / 跳过工作区模式。

## Approaches Considered

### Approach A：把每个 skill 直接复制成 plugin command（lift-and-shift）

- **Effort**: S
- **Risk**: High — 大概率跑不起来，因为 `agent:` / `workspace/...` / `ap-*` 都是悬空引用
- **Pros**: 改动最小，prompt 一字不动
- **Cons**: 用户安装后第一次运行就报错或行为偏差；后续打补丁成本反而更高
- **Reuses**: SKILL.md 文本

### Approach B：每个 plugin 一次性手写"自包含 command"，把 subagent prompt 内联，把 knowledge 文件手动拷贝

- **Effort**: M
- **Risk**: Low-Med — 一次性把架构差异处理掉，后续更新就是改 prompt
- **Cons**: AnalystPro 主仓的 SKILL.md 和 plugin commands 双轨手维护（每次主仓改一份，plugin 那份可能漂移）；knowledge 文件复制后也是两份独立，drift 风险

### Approach B+（**推荐**）：用 transformer 脚本从 AnalystPro 源生成 plugin，把"翻译规则"从文档变成可执行 contract

- **Effort**: M（写一次 transformer + 跑一次 ≈ 手写两个命令）
- **Risk**: Low — 翻译规则集中在一个脚本里，错了一处只需改脚本重跑
- **Pros**:
  - **Dual-track 问题从根上消除**：AnalystPro 主仓改 `electron/skills/<name>/SKILL.md` 或 `electron/agents/definitions/<agent>.ts` 后，跑 `npm run build:plugins` 即同步全部 plugin
  - **Knowledge drift 消除**：脚本 `cp workspace/knowledge/<files>` 到对应 plugin
  - **翻译规则有测试**：脚本本身用 vitest 测试每条规则的 input/output，比"文档约定"可靠 10 倍
  - Plugin 仍然"开箱即用"，对用户毫无差别
  - **副产物**：transformer 暴露出"AnalystPro 私有约定" vs "Claude Code 通用约定"的边界，对 AnalystPro 自身重构也有价值
- **Cons**:
  - 多写一个脚本（\~200 行 TypeScript）+ 测试
  - 实现者必须先建立 transformer，才能产出第一个 plugin（Phase 0 之后多一个 \~3 小时的步骤）
- **Reuses**: SKILL.md 文本、agent prompt 文本、knowledge 文件——**全部自动化迁移**，零手抄

### Approach C：让 AnalystPro 直接消费 plugin（plugin 成为唯一 source of truth）

- **Effort**: L
- **Risk**: Med-High — 需要改 AnalystPro 的 SkillLoader，让它能从 `analyst-plugins/` 目录加载 plugin 命令并桥接到 Secretary 调度
- **Cons**: 工作量翻倍且耦合两个生命周期；MVP 不该这么早做
- **Reuses**: 同 B+

**Recommendation**: **Approach B+**，但**诚实评估**（outside voice #4 提醒）：

- **Transformer 的真实成本**：不是 ~3 小时，是 ~1-2 个工作日（含 vitest fixture、prompt 提取的 regex 调试、`translation-rules.ts` 的迭代）
- **真实收益取决于一个变量**：AnalystPro 主仓的 SKILL.md / agent prompt **未来 12 个月会改动几次**？
  - 频繁（≥每月一次）→ transformer 一定回本
  - 偶尔（季度级）→ transformer 边际收益 = 边际成本，taste call
  - 几乎不动 → 手写 + 维护 changelog 更轻
- **Approach B+ 的逃生口**：8 个命令里 4 个无法 transformer 自动生成（interview-enricher 通用化、enrich-report 搬迁、cn-data-sources 提炼、READMEs），所以即使选 B+，仍要手写一半内容。如果 Phase 0.5 写到一半发现 transformer 价值不够 → **降级为 Approach B**：删掉 `scripts/`，所有命令手写一份，加一个 `MIRRORED-FROM.md` 列出每个命令对应的 AnalystPro 源文件。降级零摩擦。
- **当前默认选 B+** 的理由：你过去几个月每周都在迭代 deal-analyst.ts 和 SKILL.md（看 git log），属于"频繁改动"。如果未来稳定下来，可以随时降级。

### Build Transformer (`analyst-plugins/scripts/build-from-source.ts`)

这个脚本是 Approach B+ 的核心。**不是 Phase 1 的命令文件，而是 Phase 0 之后第一件事**：先有 transformer，再有第一个 plugin。

#### 翻译规则即数据

```typescript
// analyst-plugins/scripts/translation-rules.ts
export const TRANSLATION_RULES = {
  // 删除的 frontmatter 字段
  dropFields: ['agent', 'context', 'ap-type', 'ap-ui-response', 'ap-icon', 'ap-tags'],

  // 替换 model 名
  modelMap: { sonnet: 'claude-sonnet-4-6', haiku: 'claude-haiku-4-5', opus: 'claude-opus-4-7' },

  // 路径替换（按出现顺序应用）
  pathReplacements: [
    { from: 'workspace/knowledge/', to: '${CLAUDE_PLUGIN_ROOT}/knowledge/' },
    { from: 'knowledge/', to: '${CLAUDE_PLUGIN_ROOT}/knowledge/' },        // bare form
    { from: 'workspace/state/', to: './workspace/state/' },
    { from: 'workspace/inbox/', to: './workspace/inbox/' },
    { from: 'state/', to: './workspace/state/' },                          // bare form
    { from: 'inbox/', to: './workspace/inbox/' },                          // bare form
  ],

  // Subagent prompt 清洗规则（清除 Task tool 引用、统一路径）
  agentPromptCleansing: [
    { regex: /Task tool[^\n]*\n/g, replaceWith: '' },                      // 删除 Task tool 派生指令
    { regex: /派遣 [\w-]+ subagent/g, replaceWith: '直接执行' },          // 中文派遣→直接
    // path replacements 同上
  ],

  // 浏览器工具映射（AnalystPro 旧式 → Playwright MCP 命名空间）
  // ✅ Phase 0 spike VERIFIED：tool 名格式 = mcp__plugin_<plugin-name>_<server-name>__<tool>
  // ✅ 所有 3 plugin 共享 analyst-deal 的 namespace（架构决定 Q）
  // ⚠️ 4/8 AnalystPro 工具在 vanilla Playwright MCP 中不存在 ——
  //    不是 1:1 重命名而是 strategy 替换。详见 analyst-plugins-spike/phase0-findings.md。
  // 注：'to' 字段使用 ${PREFIX} 占位符；transformer 在生成时根据当前 plugin manifest 的
  //    browserToolPrefix 替换为 mcp__plugin_analyst-deal_playwright（所有 3 plugin 都用这个）
  browserToolMap: {
    'browser_navigate':      { kind: 'rename', to: '${PREFIX}__browser_navigate' },
    'browser_click':         { kind: 'rename', to: '${PREFIX}__browser_click' },
    'browser_fill':          { kind: 'rename', to: '${PREFIX}__browser_fill_form' },
    'browser_screenshot':    { kind: 'rename', to: '${PREFIX}__browser_take_screenshot',
                               note: '建议默认改用 browser_snapshot — accessibility tree 对 AI 更有用' },
    'browser_extract_text':  { kind: 'strategy', replacementText:
        '用 ${PREFIX}__browser_snapshot 抓 accessibility tree，' +
        '或 ${PREFIX}__browser_evaluate with "() => document.body.innerText"' },
    'browser_extract_links': { kind: 'strategy', replacementText:
        '用 ${PREFIX}__browser_snapshot（链接出现在树里），' +
        '或 ${PREFIX}__browser_evaluate with ' +
        '"() => [...document.querySelectorAll(\\"a\\")].map(a=>({href:a.href,text:a.textContent.trim()}))"' },
    'browser_cookie_import': { kind: 'manual', userGuidance:
        'Vanilla Playwright MCP 不支持 cookie 导入。命令需提示用户先在浏览器登录、' +
        '导出 cookies.json，命令开始用 ${PREFIX}__browser_evaluate 注入 document.cookie；' +
        '或手动跳过此命令的 cookie-gated 数据源' },
    'browser_cookie_status': { kind: 'detection-rewrite', replacementText:
        '用 ${PREFIX}__browser_navigate 跳到目标 URL；用 ${PREFIX}__browser_snapshot ' +
        '看 tree 是否包含 login form 指示词（"登录" / "login" / "sign in"）；包含 → cookie 需重新导入' },
  },
}
```

**重大修正（Phase 0 finding）**：之前 `browserToolMap` 只是字符串替换列表。实测后发现这是错的——4/8 个 AnalystPro 工具在 vanilla Playwright MCP **完全不存在**（custom-named tools），所以 transformer 必须支持 4 种策略：

| kind                | 含义                                                 | 影响命令数 |
| ------------------- | -------------------------------------------------- | ----- |
| `rename`            | 简单字符串替换（含改名如 `browser_fill` → `browser_fill_form`） | 4/8   |
| `strategy`          | 重写整段抽取逻辑为 `browser_snapshot` 或 `browser_evaluate`  | 2/8   |
| `manual`            | Plugin 无法覆盖；命令开头加用户指引（cookie 导入）                   | 1/8   |
| `detection-rewrite` | 把 cookie-status 检测改为 navigate+snapshot 启发式         | 1/8   |

Transformer 复杂度因此 +30%，但仍比手写 8 命令值——这些 strategy 文本写一次用 4 处（deal-analysis、industry-research、news-scan、browse-cn）。

**受影响命令**：deal-analysis、industry-research、news-scan、cn-data-sources knowledge file。
**不受影响**：tech-dd、memo、enrich-report、interview-notes-enricher（不用浏览器工具）。

## Failure Modes（每个命令命令开头都要处理）

每个 `commands/*.md` 在 Step 0 之前加一段 Failure Mode 检查（这段也由 transformer 注入，每个命令使用同一模板 + 命令级 override）：

```
## Failure Mode Preflight (hard-fail by default)

1. 检测必需 MCP tool 可用性：
   - WebSearch 不可用 → 致命错误，提示"Claude Code 配置异常，请检查 settings.json"
   - mcp__playwright__browser_navigate 不可用 → 降级到纯 WebSearch + WebFetch，警告"建议装 Playwright MCP 以提高中国数据源成功率"
2. 检测 ${CLAUDE_PLUGIN_ROOT} 是否解析为有效路径：
   - 用一次 `Read ${CLAUDE_PLUGIN_ROOT}/.claude-plugin/plugin.json` 试探
   - 失败（变量字面量未替换 / 路径不存在）→ **HARD FAIL**："plugin 环境变量未注入，可能 Claude Code 版本过旧或 plugin 安装损坏。请尝试 /plugin uninstall <name> && /plugin install <name>"
   - 不允许悄悄继续——之前若直接 fallback 会读到字面量路径 `${CLAUDE_PLUGIN_ROOT}/knowledge/...` 拿到 cryptic file-not-found，根因被掩盖
3. 检测必需 knowledge 文件存在（按命令 manifest 的 knowledge 列表逐一检查）：
   - 任一缺失 → **HARD FAIL**："plugin 安装可能损坏（knowledge/<name>.md 缺失）。请重装 plugin"
4. 检测当前工作目录可写：写一个 `.analyst-write-test` 后立即删除
   - 不可写 → 切换为只读模式（输出到对话窗口，不写文件），明确告知用户"当前目录不可写，仅输出到对话"
```
