# 竞对档案块采用单 schema + 公司类型三分支

为支持上市/拟 IPO 竞品的财务、芯片规格、管理层、客户深度，我们在 `knowledge/competitor_card_schema.md` **保留单一 schema**，通过「公司类型」分类把档案块分成 **上市竞品 / 拟 IPO 竞品 / 非上市创业竞品** 三档，各档有不同的必填节、jina 预算与长度上限——而**不是**拆成两/三份独立 schema 文件。原因：`portfolio-tracking` 按用户顺序拼接 N 个异构档案块进章节四，多份独立 schema 会把拼接逻辑与必填校验复杂度推给 parent；单 schema 条件式分支把该复杂度收敛在一个契约文件内，消费方只需认一个 schema。

## Considered Options

- **拆成 `competitor_card_schema_listed.md` + `_startup.md` 两份独立 schema**：结构最干净，但 parent 拼接需按类型选模板、维护两份高度重叠的字段硬规则、未来加第三档（拟 IPO）需再开一份。否决。
- **单 schema 全字段统一，所有公司都填四节**：最简单，但非上市创业公司会塞满「未公开」噪音，且与 ≤8 jina 预算不匹配。否决。
- **竞品财务统一换算到万元（对齐 `financial_ratios.md` 投后公司口径）**：可提升跨公司可比性，但竞品财报来自境内外公开披露（瑞昱/联发科为新台币、A 股为人民币），统一换算需注入一个选定汇率——直接违反 schema 硬规则「零编造」。**否决**：竞品财务表保留报告原币种 + 明标单位，不做汇率换算；可比性由分析师在小结环节人工处理。

## Consequences

- 财务节走轻量独立表，**不复用** `financial-analyzer` 三表抽取管道（那是投后公司上传报表专用，口径为万元）；二者刻意分离，勿合并。
- 「公司类型」默认由 enricher 首次股权 jina search 顺手判定（零额外调用），可经可选输入 `公司类型` 覆盖；`competitors.yml` / `portfolio-tracking` 刻意不加类型字段，以限制爆炸半径。
- 长度上限按档分层（上市 ≤1800 / 拟 IPO ≤1400 / 创业 ≤800 tokens），溢出压缩序固定为 产品方向段 → 管理层表 → 客户子节。
